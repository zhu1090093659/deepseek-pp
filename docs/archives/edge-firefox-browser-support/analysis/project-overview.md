## Project Overview

Task: add Edge and Firefox support to DeepSeek++ without forking the extension logic.

Date: 2026-05-25

### Architecture

DeepSeek++ is a WXT browser extension with three runtime entrypoints:

- `entrypoints/background.ts`: background orchestration, storage routing, alarms, tab orchestration, MCP runtime, and broadcasts.
- `entrypoints/content.ts`: isolated content script for DOM integration, runtime messaging, result rendering, and bridge calls.
- `entrypoints/main-world.content.ts`: main-world fetch/XHR interception and DeepSeek prompt/stream transformation.
- `entrypoints/sidepanel/`: React management UI for memory, skills, presets, automation, MCP, and settings.

Core behavior is browser-agnostic at the domain layer: memory, MCP, tool descriptors, automation scheduling, prompt augmentation, and sync are implemented under `core/`. Browser APIs are currently consumed directly at entrypoint and UI boundaries.

### Build System

- Framework: WXT 0.20.25.
- Default target: Chrome MV3.
- WXT supports browser-specific builds via `--browser` and manifest-version selection via `--mv2` / `--mv3`.
- WXT defaults Firefox to MV2, so this project must use `--browser firefox --mv3` to keep the service-worker-style architecture aligned with Chrome and Edge.

### Current Browser Surface

- Chrome/Edge: `sidePanel` permission, `side_panel.default_path`, `chrome.sidePanel.setPanelBehavior`, MV3 service worker, host permissions, native messaging.
- Firefox: should use `sidebar_action` generated by the WXT sidepanel entrypoint, must not ship Chromium-only `sidePanel` permission/API use, and requires a stable Gecko add-on ID for MV3 local/signing validation.
- MCP host permissions are runtime optional host permissions for HTTP/SSE/bridge transports.
